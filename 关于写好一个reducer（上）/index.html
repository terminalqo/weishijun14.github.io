Reducer的起源：

你所写的所有逻辑，都会被放进一个function中，然后唯一能让这些逻辑起作用的方法，就是丢给redux一个js对象，这个对象描述了发生了什么事。（dispatch(action)）这样一来，redux就会调用这些逻辑，并且传给他们一个state tree,以及一个action对象。然后，这逻辑函数就会返回一个新的state tree。并且redux会通知所有监听对象，state tree已更新。综上所述，这些包含了你所写的逻辑的function就是reducer。

Reducer基本概念：

  1. 应该是类似 (previousState, action) => newState 特征的函数
  2. 应该是纯函数：
    • 不应该有副作用（API请求，或者改变非本地的变量或者对象。）
    • 不应该调用不纯的函数（date.now, Math.Random）
    • 不应该修改入参。总是去返回一个新的更新后的对象，而不是直接去修改原始的 state tree。

Reducer基本结构：

很重要能理解你的一整个app应用中只有一个Reducer，它需要做如下的事情： - 当Reducer第一次被调用的时候，state是没有值的，所以要设置一个默认参数。 - 它需要prevState以及action来判断应该做什么事。 - 如果需要改变什么，那就返回一个全新的对象（数组）。 - 如果没有什么需要改变的，那就返回之前的state。

state基本结构:

Redux state 中顶层的状态树通常是一个普通的 JavaScript 对象。 大多数app会面对各种不同的data，通常我们把data分为三类：

  1. Domain data ：app所需要展示的数据，使用的，修改的。（从后端获取的）
  2. App state ：app状态的state，比如 第五条数据正被选中 。
  3. UI state ：ui状态，比如 弹窗显示 ，弹窗隐藏 。

应该用 Domain data 以及 App state 来定义你的state结构。而非 ui state 。

Reducer拆分：

对于一个大型应用来讲，把所有逻辑都放在一个Reducer中，将很快变得不可维护。所以，我们将把它们拆分成一小块，便于管理和维护。 如何拆分？我们将有如下三个规则：

  1. 在很多地方都有用到的，小的，可以重用的工具类。
  2. 处理一些特殊情况的函数，通常除了(state,action)之外还有别的参数。
  3. 当给定(state,action)参数时，对整个state进行更新。
